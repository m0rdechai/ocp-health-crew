📖 What is CNV HealthCrew AI?

CNV HealthCrew AI is an internal tool built by the Performance Engineering team to automate cluster health monitoring and root cause analysis for our OpenShift/KubeVirt environments. It saves us hours of manual investigation by automatically detecting issues, matching them to known Jira bugs, and providing actionable recommendations.

🤖 AI-Driven Root Cause Analysis

The --ai flag activates the full 3-level investigation pipeline. Each level digs deeper. The AI investigation loop (Level 3) is self-directed: Gemini suggests commands, the system executes them, and the AI decides when the root cause is specific enough. All issue types are investigated in parallel.

The Investigation Pipeline

L1: Pattern Match

Always runs • Instant • Rule-based

Matches failures against known_issues.json. Runs investigation commands via SSH. Determines root cause via keyword rules.

Produces:
  • Matched pattern + Jira bugs
  • Investigation command output
  • Initial root cause determination

L2: Symptom Drill-Down

When L1 finds symptom • SSH to nodes • Auto IP resolution

When a root cause rule has is_symptom: true, SSHes directly into affected nodes (auto-resolves hostname to IP) and runs targeted filesystem and diagnostic commands.

Example - DiskPressure:
  • df -h, du -sh /var/lib/containers /var/lib/kubelet /var/log
  • du -sh /var/lib/kubelet/pods/* (top pods by disk)
  • Pod UUID to workload identification via volume names

L3: AI Investigation

Gemini Pro • 3-5 rounds • Self-directed • Jira-aware

AI suggests diagnostic commands, system executes them, AI analyzes results. Repeats until the specific component is identified. Receives enriched Jira bug descriptions to compare against live symptoms.

Key properties:
  • Minimum 3 rounds before accepting final
  • Traces to specific pod/workload/namespace
  • Enriched Jira context: 29 bugs with full descriptions
  • Auto-resolves node IPs, adds core@
  • Read-only safety filter on all commands

AI Summary

With --ai-rca • Gemini Pro • Correlation

Receives ALL findings from L1-L3. Correlates failures across subsystems and identifies root cause chains.

Adds:
  • Cross-subsystem correlation
  • Root cause chain analysis
  • Confirms/challenges L1-L3 findings
  • Fills gaps the rules missed

How Pattern Matching Works (Technical Detail)

The RCA engine matches cluster failures against a knowledge base of known issues. It reads patterns from JSON files on disk, runs rule-based Python logic for root cause determination, and writes learned patterns back to JSON. No database is involved in pattern matching - it is entirely file-based.

Where the data lives

File Type Purpose
knowledge/known_issues.json JSON on disk Primary pattern store. Each entry has: pattern (keyword list), title, root_cause, suggestions, jira (bug references), inv_type, investigation_commands, source, last_matched.
knowledge/known_bugs.json JSON on disk Enriched Jira bug cache (29 bugs). Stores summary, description_snippet (300-char problem description), components (Jira component names), plus status, fix versions, affected versions. Bug descriptions are injected into the AI investigation context so the AI can compare live symptoms against known bug patterns. Refreshable via admin UI.
.learning_data.json JSON on disk Learning history. Tracks issue occurrences, recurring patterns, and learned fixes. When an issue appears 3+ times, it auto-promotes to known_issues.json with source: "learned".
_BUILTIN_SEED Python dict in knowledge_base.py 18 built-in patterns with investigation commands baked in. Used only to generate known_issues.json on first run. After seeding, all data lives exclusively in JSON.
knowledge/root_cause_rules.json JSON on disk Root cause determination rules. Each rule has: issue_types, keywords_all (AND), keywords_any (OR), cause, confidence, explanation. Editable via admin UI. Gemini can suggest new rules.

Pattern entries in known_issues.json come from 5 sources: built-in (seeded on first run), user (added via admin UI), learned (auto-promoted after 3+ occurrences), gemini (AI-suggested), and jira-scan (accepted from Jira API scans).

The 5-step pipeline

1 Failure Collection Source: live cluster data (in-memory)

analyze_failures() walks the health data dict returned by collect_data() and builds a flat list of failure objects. Each subsystem maps to a failure type:

operators.degraded → operator-degraded
operators.unavailable → operator-unavailable
nodes.unhealthy → node
pods.unhealthy → pod (one per pod)
virt_handler.high_memory → virt-handler-memory
migrations.failed → migration-failed
snapshot_issues → volumesnapshot
oom_events → oom
csi_issues → csi
etcd.unhealthy → etcd

Each failure carries: type, name, status, details, and raw_output.

2 Pattern Matching (Keyword Scoring) Source: known_issues.json

Patterns are loaded from knowledge/known_issues.json via knowledge_base.load_known_issues(). Each pattern entry has a "pattern" field containing a list of keyword strings (e.g. ["virt-handler", "memory", "rss"]).

For each failure, the engine builds a search string and scores every pattern by substring match:

failure_text = f"{type} {name} {status} {details}".lower()

for each pattern_entry in known_issues.json:
    match_count = 0
    for keyword in pattern_entry["pattern"]:
        if keyword.lower() in failure_text:
            match_count += 1
    if match_count > 0:
        candidates.append(match_count, jira_count, entry)

Pure substring match (in operator), no regex. Any match_count > 0 qualifies as a candidate.

3 Best Match Selection Writes to: known_issues.json

Candidates are sorted by two criteria: (1) most keyword matches, then (2) most Jira references as a specificity tiebreaker. The top candidate becomes the matched_issue. All candidates are kept in all_matches for the report.

If no pattern matches at all, the failure gets a generic "Unknown Issue" entry with fallback suggestions.

update_last_matched(key) writes the current timestamp to the winning pattern in known_issues.json, so the admin UI can show which patterns are actively used.

4 Deep Investigation (--ai flag) Source: known_issues.json + built-in commands in knowledge_base.py

run_deep_investigation() goes back to the cluster via SSH and runs targeted oc commands. Investigation commands come from two sources, checked in order:

1. The investigation_commands field inside the matched pattern in known_issues.json. Each command is a {cmd, desc} pair, with placeholders like {pod}, {ns}, {name} substituted at runtime. All 18 built-in patterns have investigation commands baked in.
2. For generic issue types not in the knowledge base (e.g. pod-crashloop, pod-unknown), load_investigation_commands() provides built-in command sets from knowledge_base.py.

Failures are grouped by matched_issue.title. If multiple failures share the same title, investigation runs once and results are shared across all of them (max 10 unique groups).

5 Root Cause Determination + Symptom Drill-Down Source: root_cause_rules.json

determine_root_cause() concatenates all investigation command outputs into one string, lowercases it, then evaluates rules from knowledge/root_cause_rules.json. Rules are fully configurable - editable via the admin UI, and Gemini can suggest new rules after each run.

Symptom detection: When a rule has "is_symptom": true, it triggers L2 drill-down which SSHes directly into nodes (auto-resolving hostnames to IPs) and runs targeted commands (e.g., DiskPressure SSHes to the node and runs df -h, du -sh /var/lib/kubelet/pods/*, identifies pod UUIDs via volume names). After L2, root cause rules are re-evaluated.

Example keyword checks per issue type:

Issue Type

Each match produces a (cause, confidence, explanation) tuple. Confidence is high or medium. The highest-confidence cause wins. If no keywords match, a generic cause is returned.

6 AI Investigation Loop (Level 3) Model: gemini-2.5-pro • 3-5 rounds

ai_investigate() drives an iterative loop where Gemini Pro suggests read-only diagnostic commands, the system executes them via SSH, and the AI analyzes the results. This repeats for 3-5 rounds.

Self-evaluation: The AI returns a JSON response with an is_final boolean. The system enforces a minimum of 3 rounds before accepting is_final=true, forcing deeper investigation even if the AI concludes early.
Component tracing: AI is explicitly instructed to trace resource consumption (disk, memory) back to specific pods, workloads, and namespaces. Never stops at "filesystem full" - must identify who filled it.
Directory drill-down: When large directories are found, AI recursively runs du -sh <dir>/* until the specific consumer is identified (e.g., a specific pod's container logs).
RHCOS awareness: Handles /sysroot paths, sudo requirements, and glob expansion differences on CoreOS nodes.
Safety: All AI-suggested commands pass through is_safe_command() filter. Write operations are rejected. SSH commands auto-get core@ prefix and timeouts.

All issue types are investigated in parallel using ThreadPoolExecutor(max_workers=4). Each issue type gets its own AI conversation context.

🤖 AI-Driven Root Cause Analysis - Technical Reference

The RCA pipeline has 3 levels + an AI summary layer. Each level digs deeper. L1 is always fast and free. L2 triggers on symptoms and SSHes into nodes for filesystem data. L3 uses Gemini Pro to iteratively investigate. The AI summary (Gemini Pro) correlates all findings at the end.

The 3-Level Pipeline

Level Runs when Speed What it does What it produces
L1: Pattern Match
analyze_failures() + run_deep_investigation()		 Always (with --ai or RCA flag) Instant + 10-30s SSH Matches failures against known_issues.json patterns. Runs investigation commands via SSH. Determines root cause via keyword rules in root_cause_rules.json. Matched pattern, Jira bugs, investigation output, initial root cause determination
L2: Symptom Drill-Down
determine_root_cause()		 When L1 root cause has is_symptom: true 10-20s SSH When L1 identifies a symptom (e.g., "DiskPressure"), SSHes directly into the affected node (auto-resolves hostname to InternalIP). For DiskPressure: runs df -h, breadth-first du -sh of all /var consumers, du -sh /var/lib/kubelet/pods/*, and identifies workloads via volume names in pod directories. Re-evaluates root cause rules with new evidence. Deeper evidence (e.g., disk usage breakdown, specific pod memory), refined root cause
L3: AI Investigation
ai_investigate()		 With --ai flag 30-120s per issue Gemini Pro iteratively suggests read-only commands, system executes them, AI analyzes results. Repeats 3-5 rounds with chain-of-thought reasoning. Breadth-first disk analysis, pod UUID-to-workload identification via volume names. Issues investigated in parallel (90s/AI call, 480s/thread). Specific root cause (e.g., "container registry images consumed 45GB in /var/lib/containers")
AI Summary
analyze_with_gemini()		 With --ai-rca flag ~5-10s Gemini Pro receives ALL findings from L1-L3. Correlates failures across subsystems, identifies root cause chains, confirms/challenges findings. Cross-subsystem correlation, root cause chains, severity ranking, remediation commands

Cost per run ~$0.03-0.08 (Gemini Pro for investigation and summary; 3-5 rounds/issue)
Safety All AI-suggested commands validated by is_safe_command(). Write ops rejected. SSH auto-gets core@ + timeouts.
Fallback L1/L2 always run (rule-based). L3/summary skipped if API key missing or call fails. Health check never breaks.

Data Flow

collect_data() → analyze_failures() [L1] → determine_root_cause() [L1+L2] → ai_investigate() [L3, parallel] → analyze_with_gemini() [Summary] → Report HTML

L1-L2 are rule-based and fast. L2 now SSHes into nodes for filesystem data (df -h, du -sh) and resolves node IPs automatically. L3 (ai_investigate) runs a self-directed investigation loop per issue type using Gemini Pro with chain-of-thought reasoning. The AI summary (analyze_with_gemini) uses Gemini Pro for cross-subsystem correlation. _md_to_html() converts the AI markdown response into styled HTML matching the report's dark theme.

🧠 Dynamic Knowledge Base (knowledge/)

The pattern engine loads its knowledge at runtime from JSON files in the knowledge/ directory. On first run, files are seeded from _BUILTIN_SEED in knowledge_base.py (18 patterns with investigation commands baked in). After seeding, all data lives exclusively in JSON and grows from five sources: built-in, user (admin UI), learned (auto-promoted), gemini (AI-suggested), and jira-scan.

File What it contains Used by
known_issues.json Issue patterns with: pattern (keyword list), jira (bug refs), root_cause, suggestions, investigation_commands, source, confidence, created, last_matched. Starts with 18 built-in patterns; grows via learning, Gemini, and user edits. Pattern Matching + Deep Investigation
known_bugs.json Enriched Jira bug cache: 29 bugs with summary, description_snippet, components, plus status, resolution, fix versions, affected versions. Bug descriptions are injected into AI investigation context via _get_bug_context(). Refreshable from Jira API via admin UI (now pulls rich fields). check_jira_bugs(), _get_bug_context()

Still hardcoded: AVAILABLE_CHECKS (15 check definitions in config/settings.py) defines what data to collect, not how to interpret it. The knowledge base (known_issues.json) is managed by healthchecks/knowledge_base.py with admin CRUD at /admin/knowledge.

How data flows through a build

User clicks "Run"
  |-- Build record created ------------> SQLite
  |-- Check config loaded --------------> AVAILABLE_CHECKS (hardcoded)
  |-- Thresholds loaded ----------------> .settings.json
  |
  |-- SSH + oc commands ----------------> Live cluster data (in memory)
  |
  |-- Phase 1: Pattern matching --------> knowledge/known_issues.json (dynamic)
  |-- Phase 2: Deep investigation ------> Pattern's investigation_commands + live SSH
  |-- Jira bug context -----------------> knowledge/known_bugs.json (29 enriched bugs fed to AI)
  |-- Phase 3: Gemini AI ---------------> Gemini API (external)
  |     '-- Gemini feedback loop -------> knowledge/known_issues.json (new patterns)
  |
  |-- Pattern learning -----------------> .learning_data.json
  |     '-- Promotion (3+ hits) --------> knowledge/known_issues.json
  |-- HTML report saved ----------------> reports/*.html
  '-- Build record updated -------------> SQLite
